# Режимы работы и практические сценарии

## 1. Режимы работы Kafka

### Standalone KRaft broker

Standalone KRaft broker - один процесс Kafka с ролями `broker,controller`. Он принимает клиентские подключения, хранит partitions и одновременно ведет metadata quorum из одного voter.

Технические свойства:

- `process.roles=broker,controller`;
- один `node.id`;
- один `controller.quorum.voters`;
- `replication.factor=1`;
- внутренние topics `__consumer_offsets` и transaction state log имеют replication factor 1;
- отказ процесса означает полную недоступность Kafka.

Влияние режима:

- установка и диагностика проще, чем у кластера;
- нет отказоустойчивости хранения;
- нельзя проверить реальную смену leader между brokers;
- подходит для локальной разработки и проверки Python-приложений.

Проверки:

```bash
# Broker отвечает на client listener.
"$KAFKA_HOME/bin/kafka-broker-api-versions.sh" --bootstrap-server localhost:9092

# Topic с replication factor 1 создается успешно.
"$KAFKA_HOME/bin/kafka-topics.sh" \
  --bootstrap-server localhost:9092 \
  --create \
  --topic standalone-check \
  --partitions 1 \
  --replication-factor 1
```

### Multi-broker KRaft cluster

Multi-broker KRaft cluster состоит из нескольких Kafka nodes. Каждый node может быть combined (`broker,controller`) или иметь отдельную роль (`broker` либо `controller`).

Технические свойства combined-кластера из трех узлов:

- три `node.id`: 1, 2, 3;
- одинаковый `controller.quorum.voters` на всех nodes;
- разные client listeners, controller listeners и `log.dirs`;
- topics создаются с `replication.factor=3`;
- `min.insync.replicas=2` сохраняет запись при отказе одного broker при `acks=all`.

Влияние режима:

- partitions распределяются между brokers;
- leader election переносит leadership при отказе broker;
- ISR содержит фактическое число синхронных replicas;
- consumer groups продолжают читать, если доступен leader каждой partition;
- запись с `acks=all` блокируется, если ISR меньше `min.insync.replicas`.

Проверки:

```bash
# Replicated topic.
"$KAFKA_HOME/bin/kafka-topics.sh" \
  --bootstrap-server localhost:19092 \
  --create \
  --topic cluster-check \
  --partitions 3 \
  --replication-factor 3 \
  --config min.insync.replicas=2

# Leader, Replicas и ISR.
"$KAFKA_HOME/bin/kafka-topics.sh" \
  --bootstrap-server localhost:19092 \
  --describe \
  --topic cluster-check
```

### Separated controller и broker roles

В separated mode controller nodes не хранят user topics и не принимают producer/consumer traffic. Broker nodes не участвуют в controller quorum.

Технические свойства:

- controller nodes: `process.roles=controller`;
- broker nodes: `process.roles=broker`;
- controller nodes перечислены в `controller.quorum.voters`;
- broker nodes подключаются к controller quorum через `controller.quorum.voters`;
- client traffic идет только на broker listeners.

Влияние режима:

- metadata управление изолировано от produce/fetch нагрузки;
- отказ broker не уменьшает controller quorum;
- отказ controller не обязан останавливать data plane, пока quorum сохраняется;
- конфигурация сложнее, требуется больше процессов или серверов;
- production-кластеры с высокой нагрузкой чаще используют разделение ролей.

### ZooKeeper mode

ZooKeeper mode - устаревающий режим Kafka, где metadata кластера хранится в ZooKeeper. Современные версии Kafka развиваются вокруг KRaft.

Технические свойства:

- Kafka broker подключается к `zookeeper.connect`;
- ZooKeeper хранит metadata topics, brokers, controller election;
- controller broker выбирается через ZooKeeper;
- требуется отдельный ZooKeeper ensemble.

Влияние режима:

- добавляется отдельная распределенная система для сопровождения;
- миграция на KRaft требует отдельной процедуры;
- новые installation лучше строить на KRaft;
- режим встречается в существующих production-кластерах и важен для эксплуатации legacy-сред.

## 2. Режимы доставки producer

### `acks=0`

Producer отправляет request и не ждет ответа broker.

Свойства:

- минимальная latency на стороне producer;
- producer не узнает о большинстве ошибок записи;
- возможна потеря сообщений при сетевых ошибках, недоступности leader или throttling;
- offset не возвращается приложению надежным образом.

Применение:

- неважная телеметрия с высокой частотой;
- временные debug-события;
- сценарии, где потеря части данных допустима.

### `acks=1`

Producer ждет подтверждение от leader replica.

Свойства:

- запись считается успешной после append на leader;
- follower replicas могут еще не получить запись;
- при отказе leader до replication acknowledged запись может потеряться;
- latency ниже, чем у `acks=all`.

Применение:

- события средней важности;
- логовые потоки, где допустима малая вероятность потери;
- системы, где throughput важнее строгой сохранности.

### `acks=all`

Producer ждет подтверждение от всех ISR replicas. Реальное число подтверждений контролируется `min.insync.replicas`.

Свойства:

- запись подтверждается только если достаточно ISR replicas;
- при `replication.factor=3` и `min.insync.replicas=2` кластер выдерживает отказ одного broker;
- при недостаточном ISR producer получает ошибку и может retry;
- latency выше из-за ожидания replication.

Применение:

- финансовые и бизнес-события;
- заказы, платежи, статусы доставки;
- CDC и state-changing events.

## 3. Режимы commit offset consumer

### Auto commit

`enable.auto.commit=true` фиксирует offset автоматически по таймеру.

Свойства:

- меньше кода в приложении;
- offset может быть зафиксирован до завершения бизнес-обработки;
- при сбое возможна потеря сообщения для приложения;
- подходит для простого чтения без внешних side effects.

### Manual commit после обработки

`enable.auto.commit=false`, приложение вызывает `commit()` после успешной обработки.

Свойства:

- обеспечивает at-least-once поведение;
- при сбое до commit сообщение будет прочитано повторно;
- внешняя операция должна быть идемпотентной: UPSERT, unique key, transaction id;
- это основной режим для Python consumers, пишущих в PostgreSQL.

### Commit batch

Consumer обрабатывает пачку сообщений и фиксирует offset после завершения всей пачки.

Свойства:

- выше throughput, меньше записей в `__consumer_offsets`;
- при сбое повторяется вся незакоммиченная часть batch;
- размер batch влияет на задержку и риск повторной обработки.

## 4. Режимы Kafka Connect

### Standalone mode

Один Connect worker запускается с локальным worker config и connector config.

Свойства:

- простая команда запуска;
- конфигурация connector лежит в локальном файле;
- нет автоматического failover tasks;
- подходит для разработки connector и простых single-node задач.

Влияние:

- отказ worker останавливает connector;
- перенос на другой сервер требует переноса файлов;
- offsets обычно хранятся в локальном файле или Kafka в зависимости от connector и worker config.

### Distributed mode

Несколько Connect workers с одним `group.id` образуют Connect cluster.

Свойства:

- конфиги connectors, offsets и statuses хранятся в Kafka topics;
- REST API используется для создания, обновления и удаления connectors;
- tasks распределяются между workers;
- при отказе worker tasks перебалансируются.

Влияние:

- требуется надежный Kafka cluster;
- внутренние topics Connect должны иметь replication factor 3 в production;
- изменение connector config вызывает restart/rebalance tasks;
- подходит для production-интеграций.

## 5. Режимы хранения данных

### Delete retention

`cleanup.policy=delete` удаляет старые closed segments по времени или размеру.

Свойства:

- Kafka хранит историю событий ограниченное время;
- consumer может перечитать данные только пока они не удалены;
- active segment не удаляется до закрытия;
- retention применяется на уровне partition.

Применение:

- логи приложений;
- события заказов;
- телеметрия;
- любые append-only event streams.

### Log compaction

`cleanup.policy=compact` сохраняет последнее значение для каждого key.

Свойства:

- topic превращается в журнал изменений состояния;
- старые значения key удаляются cleaner threads;
- key обязателен, messages без key не подходят для compaction-семантики;
- tombstone record с `value=null` обозначает удаление key.

Применение:

- справочники;
- текущее состояние пользователя;
- compacted changelog state store;
- таблицы конфигураций.

### Delete плюс compaction

`cleanup.policy=delete,compact` сочетает сохранение последнего состояния key и удаление слишком старых segments.

Свойства:

- compaction уменьшает число старых версий key;
- retention ограничивает возраст данных;
- используется, когда состояние key не нужно хранить бесконечно.

## 6. Технический сценарий 1: система логирования

### Архитектура

```text
Python/FastAPI service
  -> Python Kafka producer
  -> Kafka topic app-logs
  -> Python log consumer group
  -> PostgreSQL, ClickHouse или Elasticsearch
```

### Поток данных

1. HTTP-приложение обрабатывает запрос.
2. Приложение формирует структурированное log event.
3. Producer отправляет event в topic `app-logs`.
4. Key выбирается как `trace_id` или `service`, в зависимости от цели группировки.
5. Kafka хранит логовые события заданный retention period.
6. Consumer читает события, преобразует поля и записывает в хранилище поиска или аналитики.
7. Offset фиксируется после успешной записи во внешнее хранилище.

### Формат события

```json
{
  "ts": "2026-05-08T12:00:00Z",
  "service": "billing",
  "level": "ERROR",
  "message": "payment declined",
  "trace_id": "f5b6d8f2",
  "context": {
    "customer_id": "customer-1",
    "order_id": "order-77"
  }
}
```

### Настройки topic

```bash
"$KAFKA_HOME/bin/kafka-topics.sh" \
  --bootstrap-server localhost:9092 \
  --create \
  --topic app-logs \
  --partitions 6 \
  --replication-factor 3 \
  --config retention.ms=604800000 \
  --config cleanup.policy=delete \
  --config compression.type=lz4
```

Техническое влияние:

- 6 partitions позволяют параллельно писать и читать логовый поток.
- `replication-factor=3` защищает логи от потери одного broker.
- `retention.ms=604800000` хранит события 7 дней.
- `cleanup.policy=delete` сохраняет append-only поток без compaction.
- `compression.type=lz4` снижает объем диска и сети.

### Отказовые ситуации

Producer не может подключиться к Kafka:

- приложение должно логировать ошибку локально;
- при критичных логах добавляется локальный буфер;
- при некритичных логах допустим drop.

Consumer не успевает обрабатывать:

- растет consumer lag;
- добавляют consumers в ту же group, но не больше числа partitions для активного параллелизма;
- увеличивают batch обработки и оптимизируют запись во внешнее хранилище.

Внешнее хранилище недоступно:

- consumer не делает commit offset;
- после восстановления хранилища сообщения перечитываются;
- запись должна быть идемпотентной по `trace_id` и timestamp или по отдельному event id.

## 7. Технический сценарий 2: обработка заказов

### Архитектура

```text
orders-api
  -> final-orders topic
      -> billing consumer group
      -> shipping consumer group
      -> notifications consumer group
```

### Поток данных

1. `orders-api` получает HTTP request на создание заказа.
2. API назначает `order_id` и публикует событие `OrderCreated`.
3. Key выбирается как `customer_id`, если важен порядок заказов клиента, или `order_id`, если важна равномерность распределения.
4. `billing` читает `final-orders` в собственной group и публикует результат списания.
5. `shipping` читает тот же topic в другой group и создает доставку.
6. `notifications` читает тот же topic в третьей group и отправляет уведомление.
7. Каждая group имеет собственные offsets и не блокирует другие groups.

### Формат события заказа

```json
{
  "event_type": "OrderCreated",
  "event_id": "b9c8a7",
  "order_id": "order-100",
  "customer_id": "customer-1",
  "amount": 1500.0,
  "created_at": "2026-05-08T12:00:00Z"
}
```

### Режимы доставки

- `orders-api` использует `acks=all` и `enable.idempotence=true`, потому что потеря заказа недопустима.
- Consumers используют manual commit после успешного side effect.
- Side effects выполняются идемпотентно по `event_id` или `order_id`.
- Повтор события не должен создавать повторное списание или повторную доставку.

### Отказовые ситуации

`billing` упал после списания, но до commit offset:

- Kafka отдаст событие повторно после рестарта;
- billing должен проверить `event_id` в своей БД;
- если списание уже выполнено, повторная операция не выполняется, offset фиксируется.

`shipping` отстает:

- lag растет только у group `shipping`;
- `billing` и `notifications` продолжают работу;
- добавление consumers в group `shipping` повышает параллелизм до числа partitions.

Kafka broker потерян в replicated cluster:

- leader election переносит partitions на доступные brokers;
- при `replication.factor=3`, `min.insync.replicas=2`, `acks=all` запись продолжается после отказа одного broker;
- после отказа двух brokers запись может остановиться, потому что ISR меньше минимума.

## 8. Технический сценарий 3: IoT pipeline

### Архитектура

```text
sensors
  -> telemetry_generator.py
  -> telemetry topic
  -> telemetry_aggregator.py
  -> telemetry-1m topic
  -> storage/dashboard/alerting
```

### Поток данных

1. Устройство генерирует измерение: температура, давление, заряд, координаты.
2. Python producer отправляет событие в `telemetry`.
3. Key равен `device_id`, чтобы порядок событий одного устройства сохранялся в одной partition.
4. Aggregator читает `telemetry`, группирует события по `device_id` и временному окну.
5. Результат записывается в `telemetry-1m`.
6. Dashboard или alerting consumer читает агрегаты и строит состояние.

### Формат сырых событий

```json
{
  "device_id": "sensor-1",
  "ts": 1778241600.0,
  "temperature": 27.4,
  "battery": 91
}
```

### Формат агрегата

```json
{
  "window_start": 1778241600,
  "device_id": "sensor-1",
  "events": 120,
  "avg_temperature": 26.8
}
```

### Технические настройки

- `telemetry` использует много partitions для высокой скорости записи.
- `retention.ms` выбирается по сроку, за который можно восстановить агрегаты.
- `telemetry-1m` может храниться дольше, чем сырые события.
- Aggregator фиксирует offset после публикации агрегата.
- Для точной обработки окон требуется хранить состояние и учитывать late events; простая Python-реализация содержит базовый принцип, production-обработка переносится в Kafka Streams, Flink или Faust-подобный движок.

### Отказовые ситуации

Generator отправляет быстрее, чем aggregator читает:

- растет lag group `telemetry-aggregator`;
- увеличивают число partitions и instances aggregator;
- проверяют CPU, batch size, commit frequency.

Aggregator упал до публикации агрегата:

- offset не зафиксирован;
- события окна будут перечитаны;
- output topic должен выдерживать повторную публикацию через key вида `device_id:window_start` и compaction или через идемпотентную запись в downstream storage.

Поздние события приходят после закрытия окна:

- простая реализация отнесет их к текущей обработке или пропустит корректировку;
- production topology задает grace period и update/retraction semantics.

## 9. Технический сценарий 4: PostgreSQL integration через Python

### Архитектура

```text
PostgreSQL orders table
  -> pg_orders_producer.py
  -> Kafka topic pg_orders
  -> pg_orders_sink.py
  -> PostgreSQL processed_orders table
```

### Поток данных

1. Source-приложение хранит `last_id` последней прочитанной строки.
2. Каждые несколько секунд оно выполняет SQL `SELECT ... WHERE id > last_id ORDER BY id`.
3. Каждая строка преобразуется в JSON event.
4. Producer отправляет event в Kafka с key=`id` или key=`customer_id`.
5. Sink consumer читает topic.
6. Sink выполняет `INSERT ... ON CONFLICT DO UPDATE`.
7. Offset фиксируется после successful database commit.

### Ограничения polling-подхода

- удаление строк из source table не передается как событие;
- update существующей строки не будет замечен, если используется только `id > last_id`;
- `last_id` в памяти теряется при рестарте source-приложения;
- для production CDC используют Debezium или database logical replication.

### Надежность sink

Sink работает в режиме at-least-once:

- сообщение может быть обработано повторно;
- `processed_orders.id` является primary key;
- `ON CONFLICT DO UPDATE` делает повтор безопасным;
- offset commit после database commit предотвращает потерю события между Kafka и PostgreSQL.

### Проверки

```bash
# Source table.
psql "$POSTGRES_DSN" -c "SELECT count(*) FROM orders;"

# Kafka topic offsets.
"$KAFKA_HOME/bin/kafka-run-class.sh" kafka.tools.GetOffsetShell \
  --bootstrap-server localhost:9092 \
  --topic pg_orders

# Sink table.
psql "$POSTGRES_DSN" -c "SELECT count(*) FROM processed_orders;"

# Consumer group lag.
"$KAFKA_HOME/bin/kafka-consumer-groups.sh" \
  --bootstrap-server localhost:9092 \
  --describe \
  --group pg-orders-sink
```
